# 生成流程

用于从主题、资料、粗略素材生成公文。

## 目录

- 判断任务是否明确
- 选择文种框架
- 整理素材
- 搭建标题和框架
- 写首段
- 写主体
- 写结尾
- 自检
- 只给主题时的输出格式

## 1. 判断任务是否明确

先判断是否可以直接写：

- 有文种、有主题、有参考资料：直接写。
- 有参考资料但没说文种：根据资料和用途推断文种，必要时说明假设。
- 只有主题：按 `intake-and-routing.md` 进行最多 3 轮低成本询问。
- 用户要求“别问，直接写”：直接写，用 `[待补充]` 标记缺失事实。

## 2. 选择文种框架

读取 `document-frameworks.md`，选最接近的文种。

若用户提供单位模板、历史格式、上级通知要求，优先使用用户提供的格式。

若用户只要框架，不写正文，输出框架和每部分写作要点即可。

## 3. 整理素材

把材料分成：

- 上级来源：真实文件、会议、领导要求、政策依据。
- 工作措施：已经做了什么或准备做什么。
- 工作成绩：数据、变化、成果、荣誉、解决问题。
- 存在问题：短板、困难、风险。
- 下一步安排：未来措施和目标。
- 风格要求：正式、讲话、宣传、简洁、高度。

材料不足时，使用 `work-plots-and-verbs.md` 找追问方向或设置占位符。

## 4. 搭建标题和框架

先写标题和一级框架，再写正文。

质量要求：

- 同级标题句式一致。
- 大标题统领小标题。
- 一般 3-5 个大点，除非文种要求不同。
- 选择一个主逻辑：从大到小、从上到下、从虚到实、从古到今、从知到行等。

## 5. 写首段

按文种选择：

- 总结、简报、报告：时间状语 + 按 + 拿 + 推。
- 通知、方案：目的 + 依据 + 结合实际 + 现……
- 讲话：会议背景 + 为什么开会 + 引出下文。
- 新闻：时间 + 地点 + 人物 + 事件。
- 调研报告：调研背景 + 调研过程 + 引出下文。

不得编造上级文件名称。缺少时写 `[上级文件/会议精神]`。

## 6. 写主体

每个分点：

1. 先让标题表达本段主旨。
2. 措施用具体措施短语，如“开展调查摸底、推进专项整治、完善制度机制”。
3. 成绩用成效短语或结果句，如“服务效能持续提升、风险隐患有效化解、工作机制更加健全”。
4. 有数据就写数据。
5. 段尾落到成绩、要求或下一步行动。

宣传稿、通讯稿可先写故事和场景，再落到措施和成绩。

讲话稿可先写意义、背景、问题，再落到要求和措施。

## 7. 写结尾

按文种处理：

- 总结、报告：下一步表态或号召。
- 通知、方案：落实要求、报送要求、联系人、日期。
- 讲话、党课：号召。
- 新闻稿：活动效果或参会情况。
- 会议纪要：落实方向和责任。

## 8. 自检

返回前检查：

- 文种是否匹配。
- 框架是否完整。
- 上级提法是否准确或已标注占位。
- 数据是否真实来源于用户材料。
- 标题是否平行。
- 句子是否过长、过口语、过空。
- 未来安排是否写成具体措施短语，如“压实工作责任、深化专项整治、完善长效机制”。
- 成绩是否有数据或事实支撑。

## 9. 只给主题时的输出格式

```markdown
我先按[文种]处理，假设用途为[用途]、对象为[对象]。缺少事实处用[待补充]标记。

# [标题]

[正文]

待补充信息：
- [数据/文件/时间/单位/人员]
```
