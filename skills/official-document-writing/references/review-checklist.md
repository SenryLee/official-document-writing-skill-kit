# 审查清单

用于审查公文是否可用。必须先给结论，再列问题。

## 目录

- 审查结论
- 问题格式
- 通用审查项
- 分文种硬伤
- 通过标准

## 1. 审查结论

使用三档：

- `通过`：只剩轻微润色问题。
- `基本通过`：框架和事实可用，但正式使用前仍需修改。
- `不通过`：文种错误、框架缺失、事实无支撑、逻辑严重不通、政策或格式风险明显。

## 2. 问题格式

```markdown
- [严重/中等/轻微] 位置：……
  问题：……
  原因：……
  建议：……
```

按严重程度排序。

## 3. 通用审查项

### 任务和文种

- 文种是否符合用途。
- 受众是否正确。
- 是否满足上级通知、用户清单或单位要求。

### 框架

- 是否具备该文种必备部分。
- 可选部分是否过度添加。
- 标题是否平行。
- 标题和正文是否一致。

### 首段

- 总结、简报、报告是否有时间、来源、措施、结果。
- 通知、方案是否有目的和依据。
- 新闻是否有时间、地点、人物、事件。
- 上级提法是否准确。

### 材料

- 措施是否具体。
- 成绩是否有数据、案例或可观察结果。
- 问题是否具体。
- 下一步是否可执行。
- 是否残留 `[待补充]`。

### 逻辑

- 是否有清楚顺序：从过去到未来、从上到下、从大到小、从虚到实、从外到内、从知到行。
- 是否从意义跳到措施，中间缺少承接。
- 问题和整改是否一一对应。
- 结尾是否落到工作，而不是空泛抒情。

### 语言

- 措施是否主要为 `V+N`。
- 成绩是否使用 `N+V` 或结果句。
- 句子是否过长。
- 正式材料是否过口语。
- 讲话材料是否过于书面僵硬。
- 是否有重复主体、空话、废话和多余“的”。

### 事实和政策风险

- 是否疑似编造政策名。
- 是否改动上级固定提法。
- 是否有无依据数字、日期、荣誉、排名。
- 是否有需要来源支撑的政治性表述。
- 单位层级和行文权限是否匹配。

### 格式

- 编号层级是否符合单位习惯。
- 通知是否有主送、落款、日期、附件或联系人。
- 纪要是否有会议基本信息。
- 标点、书名号、引号是否一致。

## 4. 分文种硬伤

| 文种 | 常见硬伤 |
|---|---|
| 工作简报 | 理论太多、没有成绩、没有数据、亮点不集中 |
| 工作计划 | 写成过去成绩、缺上级对接、目标过细且无依据 |
| 工作总结 | 没有完成情况、只有口号、问题和下一步缺失 |
| 工作报告 | 缺形势分析或下年安排，正式程度不足 |
| 工作通知 | 对象、事项、时限、报送、联系人不清 |
| 工作方案 | 照搬上级方案，缺本级任务和责任 |
| 领导讲话 | 没有对象感，只有口号，不能从认识落到行动 |
| 调研报告 | 选题太大，缺数据，问题和建议混在一起 |
| 新闻稿 | 缺四要素，评论太多，领导排序不清 |
| 通讯稿 | 写成总结，缺场景、冲突和故事 |
| 先进事迹 | 只写品德，缺岗位成绩和数据 |
| 对照检查材料 | 没按上级问题分类，一边检讨一边写成绩，整改不对应 |
| 会议纪要 | 写成会议记录，缺决定和落实要求 |

## 5. 通过标准

不得判为 `通过` 的情况：

- 文种错误。
- 必备部分缺失。
- 关键事实无支撑。
- 上级政策提法虚化或疑似编造。
- 需要大幅重构才能使用。

`基本通过` 必须列出正式使用前必须改的事项。
