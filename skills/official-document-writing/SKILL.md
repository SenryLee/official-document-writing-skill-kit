# 公文处理

## 基本定位

把公文任务当作“行政信息处理”任务来做。先判断用户意图和文种，再选择框架，然后用句式、成分、工作剧情和审查清单完成生成、修改或审查。

支持三类任务：

1. **生成**：从主题、参考资料、粗略素材或用户要求生成公文。
2. **修改**：在保留事实和口径的基础上改结构、改语言、改标题、提高度。
3. **审查**：判断能否使用，指出硬伤、风险和修改方向。

## 低成本交互规则

优先降低用户交互成本：

- 用户要求明确、资料充分时，直接处理，不追问。
- 用户说“按我的参考资料写”“根据附件生成”“直接改这份稿子”时，直接读资料并产出。
- 信息不足但可以合理起草时，直接起草，并用 `[待补充]` 标记缺失事实。
- 只有在文种、用途或关键事实不清会导致方向错误时，才提问。
- 最多进行 3 轮交互；每轮只问 1-3 个最关键问题。
- 第 3 轮后必须进入产出，不继续追问。

优先追问顺序：

1. 先问用途和文种：给谁看、用在什么场景、是否有指定文种。
2. 再问核心事实：做了什么、有什么数据、是否有上级依据。
3. 最后问输出要求：篇幅、语气、是否需要问题和下一步。

## 资料路由

按任务读取 references：

- 判断意图、文种、是否需要提问：读 `references/intake-and-routing.md`。
- 处理标题、句式、首段、公文化语言：读 `references/writing-foundations.md`。
- 素材太少、需要补工作内容或找表达维度：读 `references/work-plots-and-verbs.md`。
- 处理段落成分、六大逻辑、提升高度、产品化、书面语和口语化：读 `references/components-and-logic.md`。
- 确定具体文种框架：读 `references/document-frameworks.md`。
- 生成初稿：读 `references/generate-workflow.md`。
- 修改已有稿件：读 `references/revise-workflow.md`。
- 审查把关：读 `references/review-checklist.md`。

需要套用结构时，使用 `assets/templates/` 中的模板。

## 不可违反

- 不编造上级文件、领导讲话、政策名称、会议名称、日期、数字、荣誉、单位事实。
- 用户提供的上级精神、固定提法、标题、数据，除非明显错误，不得擅自改写。
- 单位历史格式优先于通用模板。
- 缺少事实时，用 `[待补充]` 或简短追问，不用漂亮话填空。
- 修改任务默认做最小必要修改，不无关重写。
- 审查任务必须先给结论：`通过`、`基本通过` 或 `不通过`。
- 对照检查、述职、通知、会议纪要等有上级要求或单位格式的文种，必须优先服从给定要求。

## 默认输出

生成任务：

1. 简要说明推定文种和关键假设。
2. 必要时先给框架；用户要求直接成文时直接给正文。
3. 输出完整正文。
4. 最后列出少量 `待补充信息`。

修改任务：

1. 判断修改层级：轻度润色、结构重写、完整重写。
2. 输出修改后正文。
3. 简要说明主要改动和仍需确认的信息。

审查任务：

1. 先给结论。
2. 按严重程度列问题。
3. 给可执行修改建议；用户需要时再给修订稿。
